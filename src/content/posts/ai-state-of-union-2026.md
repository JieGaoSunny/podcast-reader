## 编者按

Simon Willison 是一个很难用单一标签定义的人。他是 Django Web 框架的联合创始人，是开源数据工具 Datasette 的作者，是一个坚持每天写博客的技术作家——但在过去两年里，他最为人知的身份是 AI 工程领域最敏锐的观察者和实践者。他发明了"prompt injection"这个术语，创造了"pelican riding a bicycle"这个著名的模型基准测试，并且正在撰写一本关于 agentic engineering patterns 的书。

这期 Lenny's Podcast 录制于 2026 年 4 月初，正值 AI coding agent 的能力在过去几个月里经历了一次质变。Simon 和主持人 Lenny 从"暗工厂"（dark factory）的概念聊起，深入探讨了 AI 如何重塑软件工程的每一个环节——从写代码到测试、从安全到产品设计、从职业规划到人类认知的极限。这不是一次充满乐观泡沫的对话，而是一个同时在享受变革红利和警惕变革风险的从业者的真实报告。

---

## 一、暗工厂：当 AI 7×24 小时替你写代码

Simon 开场就抛出了一个让人不安的词——"dark factory"（暗工厂）。这个概念来自制造业：一座完全自动化的工厂，不需要开灯，因为没有人类在里面工作。Simon 说，他观察到的最前沿团队正在构建软件领域的暗工厂：coding agent 全天候运行，自动写代码、自动测试、自动提交 PR，人类只在关键节点介入审查。

他举了一个真实案例：一家公司用开源客户端库搭建了自己的 API 模拟环境——用 Go 写的小型二进制文件模拟 Slack、Jira 等外部系统，甚至还 vibe code 出了一套假的 Slack 界面来观察 agent 的行为。这意味着 agent 可以在完全隔离的环境中开发和测试，不产生任何外部 API 调用成本。

这里最关键的创新不是代码生成本身——Claude Code、Codex 早就能做到——而是 **QA（质量保证）环节的自动化**。当 agent 产出代码后，另一组 agent 负责验证：功能正确吗？安全吗？性能可接受吗？Simon 指出，AI 在安全渗透测试领域的进步尤其惊人。OpenAI 和 Anthropic 都拥有"不对外公开的专用安全模型"，只向经过注册审核的安全研究员开放，因为这些模型已经强到可以被用来攻击网站。Anthropic 最近就用自家模型发现了 Firefox 的上百个潜在漏洞，并负责任地报告给了 Mozilla。

但 Simon 也发出了警告：这和那些用 ChatGPT 随便生成漏洞报告就提交给开源维护者的人完全不同。后者制造的是噪音，前者经过了 Anthropic 安全团队的人工验证。**工具的强大不等于使用者的可靠**——这个区分在 AI 时代越来越重要。

## 二、代码变得廉价之后，瓶颈转移到了哪里？

Simon 认为，2026 年软件工程领域最大的结构性冲击是：**写代码这件事的成本趋近于零**。

过去，一个产品经理写好 spec 交给工程团队，三周后如果幸运的话能拿到一个初版实现。现在同样的事情可能只需要三小时。这不是渐进式的提速，而是数量级的跳跃。它改变了一切：流程、组织结构、人的角色。

那么瓶颈去了哪里？Simon 的回答出人意料——**不是"想出好点子"，而是"验证好点子"**。

他分享了自己的工作方式：对任何新功能，他会让 agent 同时生成三个不同的原型实现，然后亲自试用来决定哪个方向最好。这在以前不可能——做三个原型的成本几乎等于做三个项目。现在原型几乎免费，真正昂贵的是判断力。

Simon 坦言，虽然 AI 非常擅长头脑风暴的前三分之二——把所有显而易见的想法快速铺开——但真正有价值的洞察往往出现在"再给我20个"之后的尾部，那些看似荒谬但能激发灵感的组合。你甚至可以给它一些刻意荒唐的约束："用海洋生物学的灵感来为我的 SaaS 平台设计营销策略"——大部分结果是垃圾，但偶尔会有一个火花点燃正确的方向。

至于能不能用 AI 模拟用户来做可用性测试？Simon 持怀疑态度。他认为让 ChatGPT 假装点击你的原型，得到的反馈远不如让一个真人在 Zoom 上实际操作来得可靠。**AI 能生成选项，但人类仍然是最好的裁判**——至少目前如此。

## 三、谁最危险？不是初级工程师，是中层

关于 AI 对工程师职业的影响，Simon 引用了 ThoughtWorks 最近一次高管研讨会的结论，这个结论违反了大多数人的直觉：

- **资深工程师**受益最大——AI 是他们 25 年经验的放大器，他们能用精确的工程术语和 agent 高效协作；
- **初级工程师/实习生**也受益——Cloudflare 和 Shopify 在 2025 年各招了上千名实习生，因为 AI 将实习生的有效产出时间从一个月缩短到了一周；
- **最危险的是中层工程师**——他们已经获得了初级阶段的 AI 红利，但还没有积累到资深级别的经验储备来真正发挥 AI 的放大效应。

Simon 用自己的例子说明了经验的放大效应：他能看一眼问题就知道"这是一个一句话 prompt 就能解决的 bug"还是"这是一个 agent 搞不定的深层架构问题"。这种判断力不是 AI 能给的，它来自二十五年的编程实践。

但他也承认，经验带来了一个新的认知障碍：**他对工作量的直觉完全失效了**。过去他会看到一个需求说"这得两周"然后放弃，现在同样的事情可能 20 分钟就完成——因为那两周里的大部分时间都是 AI 现在能覆盖的"手工编码劳动"。他的应对策略是不断尝试把自己认为"不可能"的任务丢给 AI，因为每次 AI 做到了，就是在更新自己的能力边界地图。

## 四、Agentic Engineering：让 Agent 写出好代码的实战模式

Simon 正在写的书叫做"agentic engineering patterns"，他在播客中分享了几个核心模式：

### Red/Green TDD：让 Agent 先跑失败的测试

Simon 坦承自己作为程序员从来不喜欢测试驱动开发（TDD）——先写测试、看它失败、再写代码让它通过——他觉得这个流程太死板。但他发现 **coding agent 是 TDD 的完美执行者**：它们不会觉得无聊，不会偷懒跳过步骤，而且用 TDD 能显著提高代码质量。

关键技巧是：在 prompt 里只需写 `red/green TDD` 五个字。Agent 知道这个术语的含义——先写测试并运行看到红色（失败），再写实现并运行看到绿色（通过）。五秒钟的 prompt 投入，换来的是 agent 不会漏测、不会写多余代码。

Simon 的一个反直觉观点是：他现在 **不再介意过度测试**。过去，100 行代码配 1000 行测试是坏信号——维护成本太高。但当维护测试本身也是 agent 的工作时，大量测试反而成了资产。代码是便宜的，测试也是。

### 从好模板开始：一个测试文件胜过千行文档

很多人会写一个详尽的 `CLAUDE.md` 文件来描述编码风格偏好。Simon 的方法更简单：**只提供一个极简的项目骨架**——一个测试文件（哪怕只测试 `1 + 1 == 2`），用你喜欢的风格写好缩进、命名、目录结构。Agent 会自动模仿这个风格。一个例子胜过一千字的描述。

### 囤积你知道怎么做的事

Simon 把这个称为"hoarding things you know how to do"——在 GitHub 上维护一个个人工具库（`simonw/tools`，193 个小型 HTML/JS 工具）和研究库（`simonw/research`，75+ 个 coding agent 驱动的研究项目）。每个项目都是一个"已验证的能力节点"：他知道某个技术能做某件事，因为 agent 已经写过代码并运行成功。

关键区别是：这不是 deep research 报告——那些只是"LLM 呕吐物"。这些是 **coding agent 实际写了代码、跑了代码、画了图的研究项目**，每一个都是可复用的起点。当新问题出现时，他可以告诉 Claude Code："去看我 research 仓库里关于 WebAssembly 和 Rust 的项目，然后用同样的方法解决这个新问题。"

## 五、工具栈：Claude Code、GPT 5.4，以及手机上的编程

Simon 目前的主力工具是 **Claude Code**——尤其是 Claude Code for Web（云端版本），因为：

1. **可以从手机操作**——iPhone 上的 Claude 应用有 Code 标签页，随时随地可以启动编码任务；
2. **安全隔离**——在 Anthropic 的服务器上运行，不怕 agent 误删本地文件；
3. **可以开 YOLO 模式**——跳过所有权限确认，同时开四个 agent 并行处理不同任务。

他强调了一个很多人忽略的点：**大量用户没能爱上 coding agent，是因为他们只用过"安全模式"**——agent 每执行一步都要请求确认，就像一个不停问"可以吗？"的学徒。关掉安全锁之后，产品体验完全不同。但前提是你知道自己在做什么，或者你在一个隔离环境里运行。

OpenAI 的 GPT 5.4 在三周前发布后让 Simon 印象深刻——性能与 Claude Opus 4.6 不相上下甚至在某些方面更好，而且更便宜。他观察到的趋势是：**各家模型在不断蛙跳式超越，但工具选择最终取决于"品味"**——一种难以言说的偏好，关乎代码风格而非对话风格。他说 Claude 写的代码更符合他的审美，可能是因为他的 prompting 风格已经和 Claude 协同进化了。

## 六、精神损耗：AI 让人更高效，也让人更疲惫

这可能是整期播客中最真实的一段。Simon 承认，尽管 AI 给了他更多时间，但 **认知疲劳前所未有**。同时运行四个 agent、在脑中维护四条任务线——到上午 11 点他就"彻底耗尽"了。

他注意到身边很多人出现了类似"赌博成瘾"的症状：睡前多开几个 agent，凌晨四点醒来检查进度。他希望这只是新鲜感带来的暂时现象——毕竟 agent 真正好用也就是最近四五个月的事——但他也担心某些公司会把"5 倍产出"变成对员工的期望值，导致系统性的倦怠。

矛盾之处在于：**这一切真的非常有趣**。Simon 把自己的新年决心从往年的"聚焦，少做事"改成了"多做事，更有野心"。他的朋友们纷纷清空了积压多年的 side project backlog——那些"总有一天要做"的项目，在几个月内全部完成了。有人甚至因此感到一种奇怪的失落：backlog 空了，接下来做什么？

## 七、Prompt Injection 与致命三角：一场尚未发生的灾难

Simon 在 2022 年——甚至早于 ChatGPT 发布——就提出了 prompt injection 这个概念。他现在有点后悔这个命名，因为它暗示了可以像修复 SQL injection 一样修复，而事实并非如此。

他提出了一个更精确的框架——**Lethal Trifecta（致命三角）**：

1. **私密信息暴露**（Private Data）——agent 能访问你的邮箱、文件等；
2. **恶意指令注入**（Malicious Input）——攻击者能通过某种渠道把指令送到 agent 面前，比如给你发一封邮件；
3. **数据外泄通道**（Exfiltration）——agent 有能力把数据发送给攻击者，比如转发邮件。

三者齐备，就构成了一个系统性安全漏洞。**唯一可靠的防御是切断三条腿中的一条**——通常是第三条（限制 agent 的数据外发能力）。试图用"更好的 AI"来检测攻击？Simon 认为即使检测率到 97%，在安全领域仍然是不及格——3% 的漏网率足以泄露全部数据。

他用 **挑战者号灾难的"偏差正常化"理论** 来类比当前的处境：每次我们冒险使用不安全的 AI 系统而没出事，就会让整个行业更加相信"没问题"——直到某天真的出大事。他已经连续三年每六个月预测一次"AI 领域的挑战者灾难"，虽然还没发生，但他认为概率在持续积累。

## 八、OpenClaw：数字宠物、安全噩梦、与不可抗拒的诱惑

Simon 对 OpenClaw 的评价充满了矛盾的欣赏：

- 从第一行代码（2025 年 11 月 25 日）到超级碗广告（2026 年 2 月）只用了三个半月——"有史以来增长最快的项目之一"；
- 它几乎完美地体现了 Simon 一直警告不要构建的东西——能访问邮箱、能代你行动的个人 AI 助手；
- 从安全角度看它是"灾难性的"——有人因此丢失了 Bitcoin 钱包；
- 但它证明了**用户对个人 AI 助手的需求强烈到愿意忽略一切风险**。

Simon 分析了 OpenClaw 成功的结构性原因：Anthropic 和 OpenAI 其实有能力构建类似产品，但 **因为不知道如何安全地构建，选择了不做**。OpenClaw 作为独立第三方没有这个约束，而且时机恰好撞上了 agent 能力的突破（新一代模型终于能可靠地调用工具了）。

他用一个朋友的比喻总结：OpenClaw 本质上是一个 **Tamagotchi（电子宠物）**，而人们购买的 Mac Mini 就是养它的"鱼缸"。

Simon 自己只在 Docker 容器里运行 OpenClaw，给了它只读的工作邮箱权限——典型的安全研究者做法。他认为"如果你能构建一个安全的 OpenClaw，那就是当前 AI 领域最大的商业机会。"

## 九、鹈鹕骑自行车：一个荒诞基准测试背后的深刻洞察

Simon 最广为人知的"发明"之一是让 LLM 生成一张 SVG 格式的鹈鹕骑自行车的图片。这不是测试图像模型，而是测试文本模型的空间推理和代码生成能力——因为 SVG 本质上是代码。

诡异的是，**鹈鹕骑自行车画得好的模型，在其他所有任务上也表现更好**。没人能解释原因，但这个相关性一直成立。各大 AI 实验室都知道这个 benchmark，GPT 5.4 mini 在最高思考级别下确实画出了最好的鹈鹕。Gemini 3.1 发布时甚至在宣传视频里放了一只鹈鹕骑自行车的动画。

Simon 承认他有"备用测试"——万一实验室专门训练鹈鹕骑自行车，他还有"豹猫骑摩托"之类的变体。但 Gemini 3.1 直接把所有动物和交通工具的组合都做了——让他"防线全破"。

这个看似玩闹的故事背后是 Simon 的核心信念：**AI 领域本质上是荒诞而有趣的**。地球上最昂贵、最耗能的计算机，你让它画鹈鹕，结果像五岁小孩画的——这本身就很好笑。保持对荒诞性的敏感和幽默感，可能是在这场变革中保持心理健康的最佳策略。

## 十、稀有鹦鹉与 AI 之年

播客的最后，Simon 没有给出任何关于 AI 未来的宏大预测。他分享了一条关于 **鸮鹦鹉（Kākāpō）** 的好消息——这种新西兰特有的不会飞的夜行鹦鹉，全球只剩 250 只，依赖 Rimu 树的果实繁殖。Rimu 树上一次大规模结果是 2022 年，四年间没有一只鸮鹦鹉幼崽出生。2026 年 Rimu 树终于再次结果，数十只幼崽已经诞生，可以通过网络摄像头观看它们孵蛋。

这或许是 Simon Willison 给出的最好建议：在被 AI 的加速度裹挟前行时，偶尔停下来看看一只罕见的绿色鹦鹉坐在巢里。**世界上有些事情仍然按照自己的节奏发生，无论模型迭代得多快。**

## 总结：一份来自前线的诚实报告

这期播客的价值不在于 Simon Willison 告诉你 AI 有多强——这已经是共识了。它的价值在于 Simon 同时处在三个位置上：他是工具的深度使用者，是安全风险的长期研究者，也是一个每天都在适应新现实的个体。他能在同一段话里说出"这太有趣了"和"这让我很担心"，而且两者都是真诚的。

几个值得带走的核心洞察：

1. **代码已经便宜到接近免费**——但判断力、品味和经验变得更加昂贵；
2. **中层工程师是最危险的位置**——不是因为 AI 取代了他们，而是因为 AI 既没有给他们初级阶段的启动加速，也没有放大他们尚未积累的深层经验；
3. **安全问题不会被"更好的 AI"解决**——致命三角是结构性的，唯一可靠的防御是在系统设计层面切断攻击路径；
4. **要在 AI 时代积累价值，就要"囤积"已验证的能力节点**——不是收藏文章，而是让 agent 写代码、跑代码、产出可复用的研究成果；
5. **保持野心，保持幽默，保持对荒诞的欣赏**——这可能是在变革中不被压垮的最好方式。

Simon 在整个对话中反复强调的一个词是 **agency**——人类的能动性。讽刺的是，AI agent 恰恰没有 agency。它们没有动机、没有好奇心、不会在半夜想到一个荒唐的点子然后兴奋得睡不着。这些仍然是人类独有的。

至少在 2026 年的 4 月，这仍然是我们最大的优势。
